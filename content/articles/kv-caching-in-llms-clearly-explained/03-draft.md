# LLM 中的 KV 缓存，清晰解释

你一定每次使用 ChatGPT 或 Claude 时都注意到第一个 token 出现明显更慢。然后其余的几乎瞬间流出。

幕后是一个叫做 KV 缓存的刻意工程决策，目的是让 LLM 推理更快。

在我们进入技术细节之前，这里是 LLM 推理有 KV 缓存和无 KV 缓存的并排对比：

现在让我们从头理解它是如何工作的。

## 第一部分：LLM 如何生成 token

Transformer 处理所有输入 token 并为每个 token 生成一个隐藏状态。那些隐藏状态被投影到词汇空间，产生 logits（词汇表中每个词一个分数）。

但只有最后一个 token 的 logits 重要。你从中采样，得到下一个 token，附加到输入，然后重复。

这是关键洞察：要生成下一个 token，你只需要最近 token 的隐藏状态。每个其他隐藏状态都是中间副产品。

## 第二部分：注意力实际计算什么

在每个 Transformer 层中，每个 token 得到三个向量：查询（Q）、键（K）和值（V）。注意力将查询与键相乘得到分数，然后用那些分数加权值。

现在只关注最后一个 token。

QK^T 的最后一行使用：

- 最后一个 token 的查询向量
- 序列中所有的键向量
那一行的最终注意力输出使用：

- 相同的查询向量
- 所有的键和值向量
所以为了计算我们需要的唯一隐藏状态，每个注意力层需要来自最新 token 的 Q，以及来自所有东西的 K 和 V。

## 第三部分：涉及的冗余

生成第 50 个 token 需要 token 1 到 50 的 K 和 V 向量。生成第 51 个 token 需要 token 1 到 51 的 K 和 V 向量。

Token 1 到 49 的 K 和 V 向量已经计算过了。它们没有改变。相同的输入，相同的输出。但模型在每一步都从零重新计算它们。

这就是每步 O(n) 的冗余工作。在整个生成过程中，O(n²) 的浪费计算。

## 第四部分：修复

不再在每一步重新计算所有 K 和 V 向量，而是存储它们。对于每个新 token：

1. 只为最新 token 计算 Q、K 和 V。
2. 将新的 K 和 V 追加到缓存。
3. 从缓存中检索所有之前的 K 和 V 向量。
4. 用新的 Q 对完整缓存的 K 和 V 运行注意力。
这就是 KV 缓存。每步每层一个新的 K 和一个新的 V。其他一切都来自内存。

注意力计算仍然随序列长度缩放（你要关注所有键和值）。但产生 K 和 V 的昂贵投影每 token 只发生一次，而不是每步发生一次。

## 第五部分：首 token 时间

现在你可以看到为什么第一个 token 很慢。

当你发送提示时，模型在一个前向传递中处理整个输入，为每个 token 计算和缓存 K 和 V 向量。这是预填充阶段，是请求中计算最密集的部分。

一旦缓存预热，每个后续 token 只需要一个 token 的单一前向传递。

那个初始延迟叫做首 token 时间（TTFT）。更长的提示意味着更长的预填充，意味着更长的等待。优化 TTFT（分块预填充、投机解码、提示缓存）本身是一个深入的话题，但动态总是相同的：构建缓存是昂贵的，从缓存读取是便宜的。

## 第六部分：权衡

KV 缓存用计算换内存。每层为每个 token 存储 K 和 V 向量。对于 Qwen 2.5 72B（80 层、32K 上下文、隐藏维度 8192），单个请求的 KV 缓存可能消耗几 GB 的 GPU 内存。在数百个并发请求时，它经常超过模型权重本身。

这就是为什么存在分组查询注意力（GQA）和多查询注意力（MQA）：跨查询头共享键/值头，减少内存，最小化质量损失。

这也是为什么加倍上下文长度是困难的。加倍窗口，加倍每请求的 KV 缓存，更少的并发用户。

## tl;dr

KV 缓存在自回归生成过程中消除冗余计算。之前的 token 总是产生相同的 K 和 V 向量，所以你计算一次并存储它们。每个新 token 只需要它自己的 Q、K 和 V。然后，注意力对完整缓存运行。

实际上 5 倍加速。代价是 GPU 内存，这在大规模时成为绑定约束。每个 LLM 服务堆栈（vLLM、TGI、TensorRT-LLM）都建立在这个想法之上。
