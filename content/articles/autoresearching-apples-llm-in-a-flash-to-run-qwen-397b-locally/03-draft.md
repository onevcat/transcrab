# 用 Autoresearch 研究苹果的 "LLM in a Flash" 来本地运行 Qwen 397B

几天前[我发推特说我在用 Claude Code 做一个实验](https://x.com/danveloper/status/2033940227736100873)，看能不能在我的 MacBook Pro 上运行 Qwen 3.5 397B。不是蒸馏版或小模型，而是真正的 3970 亿参数专家混合模型，在磁盘上占 209 GB，在一台 48 GB 内存的机器上。

自从三年前我读了苹果的["LLM in a Flash"](https://arxiv.org/pdf/2312.11514) 论文以来，我一直在等待这成为可能。论文为运行超过 DRAM 容量的模型提出了一个清晰的论点，从闪存流式传输权重，而且苹果一直在推出看起来就是为这种工作负载量身定制的硬件。我一直在等待苹果做点什么，但他们就是没有。在本地运行真正大的模型一直是我的一个白鲸，因为本地 AI 非常吸引人，但较小的模型就是不够聪明，而且它们仍然把 GPU 和内存压得很重。我想要的是在自有硬件上运行的前沿级智能，而且我一段时间以来一直认为苹果的架构应该使其成为可能。

关键是，我个人从来不够聪明，无法自己完成这种工作。Metal 着色器、Objective-C 推理引擎、低级 I/O 优化，这些都不是我的强项。但时机终于与 AI 生态系统中使这成为可能的一切结合起来了。Opus 4.6 非常非常聪明，Claude Code 使代理工程成为我们现在真正可以做的事情。Anthropic 让工具调用可以似乎永远循环运行，这意味着你可以把一个 Agent 指向一个难题，让它自主迭代。Karpathy 的 autoresearch 方法正好在正确的时间出现，让我能够与 Claude 一起解决这个问题，并使其成为真正可以成为现实的东西。所有的星星都对齐了，结果发现给 Opus 一个要优化的指标和一个"永远不要停直到达到这个数字"的目标就是缺失的那一块。

最终它以每秒 5.7 个 tokens 的持续速度运行，最大 token 吞吐量 7.07，同时保持生产质量的输出，使用约 5.5 GB 常驻内存。以这些速度，硬件本身几乎没有被动用，这让我相信我们还有更多潜力可以挖掘。

论文和代码在 [https://github.com/danveloper/flash-moe](https://github.com/danveloper/flash-moe)。

在我继续之前，我想把一件事说清楚，因为这对于理解这类工作现在是什么样子很重要。我没有写任何代码。不是 Objective-C 推理引擎（约 5000 行），不是 Metal 着色器（约 1100 行），不是 2 位再量化管道，不是测试。Claude 写了所有这些。我的角色是给它想法和正确的参考材料（LLM in a Flash 论文、@maderix 的精彩文章，他最近逆向工程了苹果神经引擎，以及 Karpathy 的 autoresearch 仓库），然后放开它，一旦它达到研究的某个阶段并开始遇到瓶颈，就进来与它合作进行性能优化并将其推向下一个水平。这种协作模式——Agent 做实现工作，人类在瓶颈点提供方向和系统级洞察——最终成为一种非常有效的 工作方式。整个旅程大约 24 小时和 90 个实验，其中 42% 的实验被丢弃。失败和成功一样有信息量。

我想谈谈系统是如何工作的，但我认为更有趣的故事实际上是关于我运行它的硬件……

苹果为追求"轻薄"所做的每一个设计决定最终都帮助了我们正在尝试做的事情。当他们把一切移到单个芯片上时，他们用铜把 CPU、GPU 和 SSD 控制器连接在一起。他们这样做是因为它使笔记本电脑更薄更省电，但也意味着存储和 GPU 之间的数据没有总线跳跃成本。我的 M3 Max 从 SSD 做 17.5 GB/s 顺序读取，比苹果在他们自己的"LLM in a Flash"论文中在 M1 Max 上测量的快 3 倍，而且它全部通过统一内存架构流动，CPU 和 GPU 共享相同的物理内存空间。最重要的是，沿途到处都有硬件缓存。这个项目最终变成的大部分是一个缓存优化问题，我们发现的最重要的优化是停止尝试用我们自己的缓存来超越硬件的缓存。

MoE 模型最终对这种方法效果非常好，因为推理时它们的稀疏性令人难以置信。Qwen 3.5 397B 每层有 512 个专家，但每个 token 只激活 10 个，我们发现可以剪枝到 4 个而没有质量下降（K=3 导致质量立即崩溃，这表明路由学会将关键计算分配到正好 4 个专家）。这意味着对于任何给定 token，只需要不到 2% 的专家权重，而且我们能够以可忽略的质量损失对专家权重进行 2 位再量化（每层 RMSE 0.001 到 0.003），这将专家存储从 209 GB 减少到 120 GB。所以你最终得到相对较少的相对较小的权重块，可以从磁盘为每个 token 流式传输，如果你能足够快地将它们传送，你就永远不需要把模型保存在内存中。

话虽如此，MoE 模型中的非确定性路由实际上使缓存优化比密集架构更难，因为你不知道下一层需要哪些专家。一个厚的密集模型，下一层权重总是可预测的，可能对这种流式推理更好，因为你实际上可以有效预取。值得探索。

推理引擎本身用 Objective-C 和 Metal 着色语言编写，热路径中没有任何 Python，没有 ML 框架。它从磁盘流式传输专家权重，通过融合的三命令缓冲区 GPU 管道运行，CPU 在 GPU 还在计算当前层结果时加载下一层的专家。

整个项目最反直觉的发现是，删除精心设计的 9.8 GB Metal LRU 专家缓存，让 macOS 自己处理缓存，使一切快了 38%。Claude 在 GPU 可见的共享内存中构建了一个复杂的应用级缓存，但它实际上在伤害我们。发生的事情是 Metal 缓存的 GPU 可见页面强制苹果的硬件内存压缩器以每秒 60000 到 130000 次解压的速度持续工作，仅在日常维护上就燃烧了 1 到 2 GB/s 的内存带宽。删除缓存，压缩器安静下来，解压下降到接近零，所有带宽都可以用于有用的工作。这与 PostgreSQL 文档教导的不要构建与应用层缓冲区池竞争 OS 缓冲区缓存的课程相同，最终成为整个项目的主题：信任硬件，让软件靠边站。

这个系统的理论吞吐量下限，仅受 SSD 带宽限制，是 18.6 tok/s。我们达到了 5.7，硬件几乎没有出汗，这意味着还有很大的提升空间。M4 Max 应该有约 25 GB/s SSD 带宽，让你无需任何软件更改就达到约 8 tok/s，而且苹果的 SSD 性能每代提高约 20%。在 2 到 3 代硬件之内，在笔记本电脑上运行 4000 亿参数的模型达到 10+ tok/s 就是基准。

这种方法也可以推广到 Qwen 之外。671B、37B 活跃的 DeepSeek-V3 是一个明显的候选者，而且实际上任何专家权重占主导地位的 MoE 模型都可以用这种方法。

苹果三年前发布了"LLM in a Flash"，告诉所有人 SSD 是一个完全可行的权重存储，然后他们持续推出的硬件正好证明了这一点。SSD 性能每年都在提升，我希望我们将开始看到来自开源实验室的越来越多前沿模型，它们就是，在你的电脑上运行。我非常感谢 MLX 开源社区为利用苹果硬件的 AI 潜力所做的持续努力。没有 MLX 作为起点，这是不可能发生的。我看好苹果。他们在 AI 有一个秘密武器，我们才刚刚开始看到它的潜力。
